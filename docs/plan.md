# План реализации JaCoCo Coverage Aggregator

## Цель
Создать систему для периодического сбора, агрегации и публикации данных о покрытии кода нескольких Java‑сервисов (микросервисов) в runtime с поддержкой версионирования.

## Основные требования
1. Поддержка множества сервисов, каждый из которых может иметь несколько версий.
2. Каждая версия сервиса может иметь несколько инстансов (реплик).
3. Периодический опрос каждого инстанса через `jacococli dump`.
4. **Автоматическое определение версии инстанса** (через HTTP‑эндпоинт или из конфигурации).
5. Агрегация данных покрытия по всем инстансам **в разрезе версий**.
6. Генерация отчётов в формате HTML **для каждой версии** с помощью `jacococli report`.
7. Публикация отчётов через встроенный Go‑веб‑сервер (роутер Chi) с навигацией по версиям.
8. **Динамическое обновление списка инстансов через REST API**.
9. **Экспорт метрик в Prometheus** (процент покрытия, количество строк и т.д.) с labels service, version.
10. Контейнеризация (Docker) для простого развёртывания.
11. Конфигурация через YAML‑файл.

## Архитектурные компоненты

### 1. Конфигурация
**Формат**:
```yaml
services:
  - id: service-a
    name: "Service A"
    # Шаблон URL для загрузки исходников (опционально)
    sources_url_pattern: "http://repo.com/xxx/{{service}}/{{version}}/service-version-source.jar"
    # Список версий с путями к исходникам и class‑файлам
    versions:
      - version: "1.0.0"
        source_path: /sources/service-a/1.0.0
        class_path: /classes/service-a/1.0.0
      - version: "1.1.0"
        source_path: /sources/service-a/1.1.0
        class_path: /classes/service-a/1.1.0
    # Инстансы (версия может быть указана явно или определена автоматически)
    instances:
      - host: 10.0.0.1
        port: 6300
        version: "1.0.0"   # если не указана – будет определена через /version эндпоинт
      - host: 10.0.0.2
        port: 6300
        # version не указана – будет запрошена у инстанса
    poll_interval: 60s
    version_detection:
      enabled: true
      endpoint: "/version"   # HTTP‑путь, возвращающий версию в plain text
      timeout: 5s
  - id: service-b
    name: "Service B"
    versions:
      - version: "2.1.0"
        source_path: /sources/service-b/2.1.0
        class_path: /classes/service-b/2.1.0
    instances:
      - host: 10.0.0.3
        port: 6300
    poll_interval: 30s
```

**Загрузка конфигурации**:
- Пакет `internal/config` – чтение YAML, валидация, предоставление структурированных данных.
- **Динамическое обновление**: конфигурация хранится в памяти и может быть изменена через REST API.

### 2. Сборщик (Collector)
**Ответственность**:
- Для каждого инстанса из конфигурации выполнять `jacococli dump` с заданным интервалом.
- **Определять версию инстанса** (если не указана в конфигурации) через HTTP‑запрос к эндпоинту `/version` (или другому, заданному в `version_detection`).
- Сохранять полученный `.exec`‑файл в директорию `./data/{service_id}/{version}/runtime-jacoco/hosts/{instance_id}/jacoco.exec` (перезаписывает предыдущий).
- Логировать успешные сборы и ошибки.

**Реализация**:
- Пакет `internal/collector`.
- Использование `exec.Command` для вызова `java -jar jacococli.jar dump ...`.
- HTTP‑клиент для запроса версии (с таймаутом).
- Конкурентный опрос инстансов (горутины + пул воркеров).
- Обработка таймаутов и повторных попыток.

### 3. Генератор отчётов (Reporter)
**Ответственность**:
- Для каждого объединённого `.exec`‑файла (per version) генерировать отчёт (HTML) с помощью `jacococli report`.
- Использовать пути к исходному коду и class‑файлам из конфигурации для соответствующей версии.
- Сохранять отчёты в директорию `./data/{service_id}/{version}/runtime-jacoco/html-report/` (для runtime‑покрытия) и `./data/{service_id}/{version}/unit-jacoco/html-report/` (для unit‑покрытия).

**Реализация**:
- Пакет `internal/jacoco` (вместе с Collector и функциями merge).
- Вызов `jacococli report` с соответствующими аргументами.

### 4. Публикатор (Publisher)
**Ответственность**:
- Запускать встроенный Go‑веб‑сервер на роутере Chi.
- Раздавать статические файлы отчётов из `./data/` по HTTP, с маршрутизацией по типам отчётов (unit‑jacoco/html‑report/, runtime‑jacoco/html‑report/, full‑jacoco/html‑report/).
- Раздавать встроенный frontend (SPA) из Go‑embedded файловой системы (`go:embed`).
- Предоставлять REST API для получения данных о покрытии.
- Отображать индексную страницу с навигацией по сервисам и версиям.
- **Предоставлять эндпоинт `/metrics` в формате Prometheus**.

**Реализация**:
- Пакет `internal/publisher`.
- Использование `github.com/go-chi/chi` для маршрутизации.
- Фронтенд собирается при сборке (`make build`) и копируется в `internal/publisher/frontend/assets/`, затем встраивается в бинарник через `go:embed`.
- **REST API**:
  - `GET /api/status/table` – получить таблицу статусов всех сервисов и версий.
  - `GET /api/coverage/{service}/{version}?type=unit|runtime|full` – получить данные о покрытии.
- **Prometheus Metrics**:
  - `GET /metrics` – метрики покрытия (процент покрытия, количество покрытых/непокрытых строк) с labels `service`, `version`, `type`.
  - Метрики системы (количество инстансов, успешные/неуспешные опросы).
- Эндпоинты для отчётов:
  - `GET /` – индексная страница (SPA).
  - `GET /reports/{service}/{version}/unit/*` – отчёт unit‑покрытия.
  - `GET /reports/{service}/{version}/runtime/*` – отчёт runtime‑покрытия.
  - `GET /reports/{service}/{version}/full/*` – отчёт полного покрытия (unit + runtime).

### 5. Управление инстансами (Instance Store)
**Ответственность**:
- Хранение списка инстансов в JSON‑файле (`instances.json`).
- Управление статусами опроса инстансов (in‑memory).
- Предоставление API для добавления, получения и удаления инстансов.

**Реализация**:
- Пакет `internal/instance`.
- `Store` – хранилище инстансов с персистентностью в JSON файл.
- `StatusManager` – управление статусами опроса (polling, success, error).
- При старте приложения выполняется миграция инстансов из YAML конфига в `instances.json`.
- Runner использует инстансы только из хранилища (единый источник истины).

**REST API**:
- `POST /api/instances` – добавление инстанса.
- `GET /api/instances` – получение списка инстансов.
- `GET /api/instances/status` – получение статусов опроса инстансов.

### 6. Хранилище (Storage)
**Ответственность**:
- Организация файловой структуры для сырых данных, объединённых файлов и отчётов с учётом версий.
- Предоставление единого интерфейса для доступа к файлам данных (сырые `.exec`, объединённые файлы, отчёты) другим компонентам (сборщик, агрегатор, генератор отчётов, публикатор).

**Структура данных**:

Хранилище организует данные в виде иерархии, отражающей логическую группировку по сервисам, версиям и типам данных. Каждый сервис (`service_id`) содержит одну или несколько версий (`version`). Для каждой версии хранятся:

- **Исходные тексты и скомпилированные классы** – кэшированные файлы, необходимые для генерации отчётов.
- **Данные покрытия unit‑тестов** (baseline) – `.exec`‑файлы и HTML‑отчёты, полученные при запуске unit‑тестов.
- **Данные покрытия runtime** – `.exec`‑файлы, собранные с работающих инстансов, и агрегированные отчёты.

Физическая структура директорий внутри корня `/data`:

```
/data
  /{service_id}
    /{version}
      /sources                     # Исходные тексты (кэш)
      /classes                     # Скомпилированные классы (кэш)
      /unit-jacoco
        /jacoco.exec               # .exec‑файл от unit‑тестов (baseline)
          jacoco.exec
        /html-report               # HTML‑отчёты unit‑тестов
      /runtime-jacoco
        /hosts                     # Последний .exec‑файл для каждого инстанса (перезаписывается)
          /{instance}/jacoco.exec
        /jacoco.exec               # Копия latest для версии
          jacoco.exec
        /html-report               # HTML‑отчёты runtime‑покрытия
      /full-jacoco                 # Полное покрытие (unit + runtime)
        /jacoco.exec               # .exec‑файл после слияния unit и runtime
          jacoco.exec
        /html-report               # Итоговый HTML‑отчёт
          index.html
          jacoco-resources/
      /meta
        last_updated.txt
```

**Генерация full‑отчёта**:
- При инициализации: если есть `runtime-jacoco/jacoco.exec` (данные с предыдущих прогонов), они объединяются с `unit-jacoco/jacoco.exec` через `jacococli merge`, результат сохраняется в `full-jacoco/jacoco.exec` и генерируется HTML‑отчёт.
- Если runtime данных нет при инициализации, используется только `unit-jacoco` для full‑отчёта.
- При каждом runtime сборе (после `jacococli dump`): запускается `generateFullReport`, который объединяет unit + runtime и генерирует full‑отчёт заново.

**Примечания**:

- Директории `sources` и `classes` заполняются либо из локальных путей, указанных в конфигурации (`source_path`, `class_path`), либо загружаются по URL (если задан `sources_url_pattern`).
- `unit-jacoco` используется для baseline‑покрытия (например, из CI). Эти данные не агрегируются с runtime‑покрытием, но могут отображаться отдельно в UI.
- `runtime-jacoco/hosts` содержит последний `.exec`‑файл для каждого инстанса (перезаписывается при каждом dump). Имена поддиректорий соответствуют идентификаторам инстансов (например, `instance-1`, `instance-2` или `host:port`).
- `runtime-jacoco/jacoco.exec` – копия latest файла для версии (используется для генерации отчётов).
- `runtime-jacoco/html-report` – итоговые HTML‑отчёты, сгенерированные на основе `.exec`‑файла.
- `full-jacoco` содержит итоговое покрытие, объединяющее unit‑ и runtime‑данные (слияние `unit-jacoco/jacoco.exec` и `runtime-jacoco/jacoco.exec`). Используется для генерации сводного отчёта, отражающего полное покрытие кода.
- `meta` хранит служебную информацию: время последнего обновления.

Такая структура позволяет:
- Чётко разделять данные unit‑ и runtime‑покрытия.
- Хранить исходные тексты и классы в версионированном виде.
- Хранить только последний файл (без истории).
- Поддерживать генерацию отчётов без перемешивания с другими версиями.

**Реализация**:
- Пакет `internal/storage`.
- Основные методы:
  - `SaveRuntimeExec(sv ServiceVersion, host string, port int, data []byte) error` – сохранение `.exec`‑файла в `runtime-jacoco/hosts/{host_port}/jacoco.exec` (перезаписывает предыдущий).
  - `SaveBaselineExec(sv ServiceVersion, data []byte) error` – сохранение baseline `.exec`‑файла в `unit-jacoco/jacoco.exec`.
  - `GetBaselineExec(sv ServiceVersion) ([]byte, error)` – чтение baseline файла.
  - `CopyLocalSources(sv ServiceVersion, srcPath string) error` – копирование исходников из локального пути.
  - `CopyLocalClasses(sv ServiceVersion, classPath string) error` – копирование классов из локального пути.
  - `DownloadAndExtract(sv ServiceVersion, subdir, url string) error` – скачивание и распаковка из URL.
- Использование стандартных библиотек Go: `os`, `path/filepath`, `io`, `archive`.
- Поддержка конкурентного доступа через `sync.RWMutex` или каналы.

**Конфигурация**:
Параметры хранилища задаются в основном конфиге.

**Обработка ошибок и логирование**:
- Логирование операций записи/чтения (уровень debug).
- Обработка ошибок файловой системы (отсутствие прав, недостаток места) с возвратом понятных ошибок.
- Мониторинг свободного места на диске, предупреждение при достижении лимита.

**Тестирование**:
- Модульные тесты с использованием временной директории (`testing.T.TempDir`).
- Табличные тесты для проверки корректности путей и именования файлов.
- Интеграционные тесты на реальной файловой системе (проверка retention).

**Интеграция с другими компонентами**:
- **Сборщик** использует `SaveRuntimeExec` для сохранения `.exec`‑файла.
- **Генератор отчётов** использует `jacococli report` для генерации HTML.

### 7. Основной цикл
Основной цикл реализован в пакете `internal/runner`.
- Использование `time.Ticker` для периодического выполнения.
- Конфигурируемые интервалы (по умолчанию: сбор каждые 5 минут).
- Упрощённый цикл: **dump → report** (без merge, так как данные уже агрегированы - один файл на инстанс).
- При необходимости: копирование `runtime-jacoco/hosts/{instance}/jacoco.exec` → `runtime-jacoco/jacoco.exec`.

### Основной цикл
- **Язык**: Go 1.21+
- **Веб‑фреймворк**: Chi
- **Конфигурация**: YAML (библиотека `gopkg.in/yaml.v3`)
- **Логирование**: `slog` (стандартная библиотека)
- **Метрики Prometheus**: `github.com/prometheus/client_golang`
- **Встраивание фронтенда**: `go:embed` (статика собирается при `make build` и встраивается в бинарник)
- **Контейнеризация**: Docker, многостадийная сборка
- **Инструменты JaCoCo**: `jacococli.jar` (скачивается при сборке образа)

## Масштабирование (Scale Considerations)
- **Конкурентность**: Использование горутин для параллельного опроса множества инстансов.
- **Ограничение ресурсов**: Настройка пула воркеров, чтобы не перегружать сеть/CPU.
- **Динамическое управление**: REST API позволяет адаптироваться к изменениям в кластере без перезапуска.
- **Распределённость**: В будущем можно вынести сборщиков в отдельные подсистемы (sidecar‑агенты).
- **Хранение**: При большом объёме данных可以考虑 использовать объектное хранилище (S3) для `.exec`‑файлов.

## Риски и митигации
- **Зависимость от JaCoCo CLI**: Если CLI меняется, может сломаться совместимость. Митигация: тестирование на нескольких версиях JaCoCo.
- **Сетевые проблемы**: Таймауты и повторные попытки должны быть настроены адекватно.
- **Объём данных**: Большое количество инстансов и версий может привести к исчерпанию дискового пространства. Митигация: ручная очистка.
- **Консистентность конфигурации**: Динамические изменения должны быть атомарными и потокобезопасными. Митигация: использование мьютексов или каналов.
- **Определение версии**: Если инстанс не отвечает на `/version`, сбор покрытия может быть невозможен. Митигация: fallback на версию из конфигурации, логирование предупреждений.
